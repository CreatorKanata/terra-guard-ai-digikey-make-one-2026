# FRDM-MCXN947 開発用スキル — 完全AI開発を支える仕組み

本作 TerraGuard AI は、**人間がコードを 1 行も書かず**、ビルド・書き込み（flash）・学習コマンドの実行まで
**すべて AI コーディングエージェント（Claude Code）がハンドリング**して開発しました。
それを成立させている中核が、リポジトリに同梱した **Claude Code スキル**です。

> 「スキル」とは、Claude Code（AIエージェント）に**特定作業の確定手順・確定パス・ハマりどころ**を
> 事前に教え込んでおく仕組みです。AI は作業の着手時にスキルを起動するだけで、
> 毎回ブレずに同じ手順で実機開発を進められます。

このドキュメントは、**他の参加者がそのまま学べる／再現できる**よう、スキルの中身と狙いを解説します。

---

## 1. なぜスキル化したのか

組み込み／エッジAI開発は、**環境依存の“地雷”**が多い分野です。

- ビルドツールやプローブの**絶対パス**（LinkServer のバージョン、venv の場所…）
- センサが ACK しない**クロック設定**、LED が光らない**GPIOクロックゲート**
- 起動直後に無音ハングする **SysTick の罠**、NPU が回らない**変換ツールのバージョン不一致**

これらは「一度ハマって解決した」ことを忘れると、AI でも人でも**毎回同じ穴に落ちます**。
そこで、解決済みの確定手順・確定パス・症状と対処を**スキルに集約**し、
**CLAUDE.md からは「作業前に必ずこのスキルを起動せよ」とだけ指示**する構成にしました。

```text
CLAUDE.md（プロジェクト憲法）
   └─「NXP 開発は着手前に必ず frdm-mcxn947-dev スキルを起動」とだけ宣言
        └─ frdm-mcxn947-dev スキル（手順・パス・地雷の実体）
              ├─ ビルド / 書き込み / シリアル確認
              ├─ I2C 配線 / クロック / GPIO・LED / SysTick
              └─ ML / NPU ワークフロー（学習〜変換〜デプロイ）
```

この分離により、**CLAUDE.md は短く保ちつつ、AI は深い手順をいつでも正確に引ける**ようになります。

## 3. `frdm-mcxn947-dev` スキルに書いてあること

ファームと ML パイプラインに関わる操作（ビルド・flash・実機デバッグ・シリアルログ・NPU学習・データ収集）を
**すべてこのスキル経由**で行います。主な内容は以下です。

### 3-1. 確定パス（推測しない）

実機検証済みの絶対パスを列挙してあるため、AI が毎回探索せずに済みます。

- プロジェクトルート / ELF の出力先
- LinkServer の場所（バージョンは都度 `ls` で確認）
- pyserial 入り venv、シリアルポート（`/dev/cu.usbmodem*`）
- **デバッグUART ボーレート = 921600**（115200 ではない、という地雷も明記）

### 3-2. ビルド・書き込み・シリアル確認

```bash
# ビルド
cd src/FRDM-MCXN947/terra-guard-ai
cmake --preset debug      # 初回 or CMakeLists 変更時
cmake --build debug

# 書き込み（LinkServer）
/Applications/LinkServer_xx/LinkServer flash "MCXN947:FRDM-MCXN947" \
  load debug/terra-guard-ai_cm33_core0.elf
```

- `-Werror` 有効（未使用関数も即エラー）といった**ビルドの作法**まで記載。
- **flash 直後に VCOM が一時切断**される → `sleep 1` してポート再確認、という手順も明文化。
- シリアルは**テキスト行とバイナリフレームが混在**するため、`latin-1` decode ＋正規表現で抽出する読み方を指定。

### 3-3. ハマりどころ（実機で解決済み）

スキルの一番の価値はここです。**症状 → 原因 → 対処**をセットで蓄積しています。

| 症状 | 原因 | 対処 |
| --- | --- | --- |
| LED が光らない | GPIO クロックゲート未有効 | `CLOCK_EnableClock(kCLOCK_GpioX)` を呼ぶ |
| 起動直後にシリアル無音 | `SysTick_Handler` 未定義で weak default の無限ループに落ちる | ハンドラを定義する |
| センサが ACK しない | FRO12M では 1MHz SCL を作れない | FC2 のクロック源を FRO_HF_DIV に |
| センサ無反応（バスロック） | I2C バスロック | **USB 抜き差し**で回復（物理操作は人間に依頼してOK） |
| NPU 推論が回らない | converter とドライバの**バージョン不一致** | ドライバ 3.0.0 に合わせて変換 |

### 3-4. ML / NPU ワークフロー

カラス検出 2 クラス（入力 24×24×4）の**データ収集 → 学習 → int8量子化 → NPU変換 → ヘッダ生成 → 書き込み**を
1 本の手順として記載。前処理をファームと学習側で**厳密一致**させる鉄則も明記しています（詳細は [ml-model.md](./ml-model.md) / [dataset.md](./dataset.md)）。

## 4. `terra-guard-model-eval` スキル

学習したモデルが**実機で本当に使えるか**を判定するためのスキルです。

- ホールドアウト評価（混同行列 / FN / FP）は**過大評価**になりがち、という前提を共有。
- **実機ライブ背景で誤検出（confirmed crow ≈ 0）を確認**してはじめて合格、という判定基準。
- tflite（ホスト）と ファームNPU の**出力照合**で、誤検出の原因（学習 vs 変換 vs 前処理）を切り分け。
- 結果に応じた**再学習すべきかの判断**まで導く。

---

## 5. AI に任せきりにしないための約束ごと

スキルには、**人間（制作者）がやるべき物理操作**も明記しています。

- **USB ケーブルの抜き差し**（I2C バスロック回復、クリーンな起動ログ取得、文字化けの切り分け）
- はんだづけ等のハード作業

つまり、**ソフトのエンジニアリングは AI、物理は人間**という役割分担を、スキル自体が定義しています。
これにより「何を AI に任せ、何を人がやるか」が毎回ブレません。

---

## 6. 再現方法

1. リポジトリをクローンする（`.claude/skills/*/SKILL.md` が含まれています）。
2. Claude Code で開き、NXP 開発を始める前に **`frdm-mcxn947-dev` スキルを起動**する。
3. スキルに従って、AI がビルド・書き込み・シリアル確認・学習を CLI で実行する。
4. モデル検証時は **`terra-guard-model-eval` スキル**を起動する。

> パス類は環境に合わせて読み替えてください（スキル内に「都度 `ls` で確認」と書いてある箇所が手がかりです）。
